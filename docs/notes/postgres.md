- cluster - в терминал postgres это сервер (НЕ отказоустойчивая схема с несколькими нодами, вроде failover cluster и т.д.)
- иерархия объектов в postgres такая: cluster -> database -> schema -> table -> column
- в кластере может быть несколько баз, они изолированны друг от друга
- схема - это вроде неймспейса - логическое разделение таблиц
    - схема public - дефолтная схема внутри каждой базы данных
- роли в pg - это аналог пользователей. При этом если у роли есть атрибут LOGIN - с ее помощью можно подключаться к серверу (иначе - только владение объектами и выдача в ним прав - без работы с данными в таблицах). Роли создаюся на уровне кластера
    - PUBLIC - псевдо-роль, обозначает все роли в кластере
- владелец базы имеет полные права над ней. Забрать эти права напрямую нельзя - только сменив владельца базы либо удалив саму базу. Это справедливо и для владельцев других объектов. При этом можно выдать GRANT ALL на пользователя-владельца, чтобы у него не пропал доступ к объекту при смене владельца, а также как явная фиксация необходимого доступа
- search_path - параметр со списком схем для поиска объектов по неполному имени. По умолчанию имеет значение "$user", public - т.е. по умолчанию поиск сначала будет идти в схеме с именем как у текущего пользователя, потом в схеме public
- WAL - write-ahead logs - аналог trunsaction лога в mssql, используется для восстановления данных и для репликации
- по умолчанию имя сервиса - postgresql.service или postgresql@14-main.service (для конретного сервиса postgresql 14)
    - при изменениях в postgresql.conf,таких как listen_addresses,port,shared_buffers - нужно делать `restart` службы (подробно - в pg_settings.context)
        - `listen_addresses` - на каких адресах postgres слушает подключения
    - изменения в pg_hba.conf можно применять через `reload`
- postgresql.conf - задает основные параметры сервера, по умолчанию лежит в /etc/postgresql/<version>/main
- данные по умолчанию лежат в `/var/lib/postgresql/<v>/main/`
- pg_hba.conf - конфиг для указания прав на подключение к серверу
    - порядок строк важен - сравнение идет сверху вниз до первого совпадения
- типы подключений в hba
    - local - через unix-socket
    - host - сетевой через TCP с шифрованием и без
    - hostssl - аналогично host, но только с TLS-шифрованием канала
    - hostnossl - аналогично host, но только без шифрования канала
- методы подключений в hba
    - trust - без аутентификации
    - scram-sha-256 - аутентификация, с хэшем sha-256 (защищено от перехвата), рекомендуется к использованию
    - peer - для unix-socket - если подключение выполняется из под пользователя, для которого есть одноименная роль в pg, то подключение пройдет
    - md5 - так же аутентификация по хэшу, но уязвима к перехвату хэша (который можно взять из файла pg_authid) - с ним можно напрямую авторизоваться в pg
    - password - аутентификация по паролю, идет в открытом виде
    - cert - аутентификация по сертификату TLS

## Идемпотентность операций в postgresql

    - реализуется с помощью блока DO
```sql
DO $$
BEGIN
    IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'foo') THEN
        CREATE USER foo WITH PASSWORD 'bar';
    END IF;
END $$;
```
    - для команд, которые нельзя засунуть в транзакцию - `CREATE DATABASE`, `CREATE TABLESPACE`, `ALTER SYSTEM` - можно использовать фишку языка psql - \gexec
        - работает так: при запросе вида "SELECT 'CREATE DATABASE.." psql сначала отправляет на сервер SELECT, в ответ будет строка (в нашем случае - запрос SQL), и второй командой на сервер на выполнение будет отправлено значение этой строки (в нашем случае - CREATE DATABASE...). Если вернется больше строк/колонок - все они будут отправлены на выполнение
    - пример использования для проверки наличия базы перед созданием
```sql
SELECT 'CREATE DATABASE foo OWNER foo_user'
WHERE NOT EXISTS (SELECT FROM pg_database WHERE datname = 'foo')
\gexec
```
## Привилегии в postgres

- уровни объектов и операции
`DATABASE` — CONNECT, CREATE (создание схем), TEMPORARY (создавать темп-таблицы)
`SCHEMA` — CREATE (создание таблиц и т.п.), USAGE (обращение к объектам схемы)
`TABLE` / VIEW — SELECT, INSERT, UPDATE, DELETE, TRUNCATE, REFERENCES, TRIGGER
`SEQUENCE` — SELECT, USAGE, UPDATE
`FUNCTION` — EXECUTE
`COLUMN` — SELECT, INSERT, UPDATE, REFERENCES

### Шаблоны команд
`GRANT <priv> ON <object> TO <role>;`
`REVOKE <priv> ON <object> FROM <role>;`

- `DEFAULT PRIVILEGES` - для определения привилегий на новых объектах (вроде файловой маски в linux)
```sql
ALTER DEFAULT PRIVILEGES IN SCHEMA public
    GRANT SELECT ON TABLES TO readonly_role;
```
- До PG14 включительно - у PUBLIC были права на CREATE и USAGE
- С PG15 - у PUBLIC отобрали CREATE по умолчанию

## Работа в командной строке
### Проверка готовности сервера
`pg_isready` - проверяет готовность сервера принимать подключения (параметров - на локальном хосте). Код возврата: 0 — готов, 1 — отклоняет, 2 — нет ответа, 3 — ошибка
    - ключ `-h` - указать хост для проверки
    - ключ `-p` - указать порт подключения

### Команды psql
\l               # list databases — список баз
\du              # describe users — список ролей
\dn              # describe namespaces — список схем
\dt              # describe tables — таблицы текущей схемы / search_path
\dt <schema>.*   # таблицы конкретной схемы
\d <table>       # описание таблицы
\c <db>          # переключиться на базу (connect)
\conninfo        # инфо о текущем подключении
\q               # выйти
\?               # help по meta-командам
\h CREATE TABLE  # help по SQL
\i               # выполнить файл со скриптом sql (из командной строки - `psql -v ON_ERROR_STOP=1 -f init.sql` - остановиться на первой ошибке)
